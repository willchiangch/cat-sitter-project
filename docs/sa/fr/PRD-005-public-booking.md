## 背景與目的

「保母對外預約網頁」是系統最重要的流量入口。為了協助保母篩選優質客戶並自動化作業，本流程集成了登入驗證、檔期過濾、動態問卷與價格試算。飼主在此提交的資料將轉化為後台的「待評估訂單」，作為雙方服務合約的起點。

## 使用者故事

- 身為 **訪客**，我希望能透過帳號登入 (如 Google 登入或密碼) 驗證身分，以便進入保母的專屬預約網頁。
- 身為 **飼主**，我希望能自由選擇預約日期 (可選單日、連續區間、或跳著選多天)，並為這些日期綁定特定服務方案與服務次數。
- 身為 **飼主**，如果我選了某個方案，但我沒有該方案支援的寵物種類 (例如我只有貓，卻選了遛狗方案)，系統能提醒我避免下錯單。
- 身為 **飼主**，我希望在送出預約前能填寫保母要求的問卷，讓保母更精準地了解我的需求。

## 功能規格

### 主流程 (三步驟引導)

**Step 0. 登入驗證**
1. 訪客進入預約網址後，若未登入，系統強制要求進行帳號登入 (支援密碼/生物辨識/Google OAuth)。
2. 登入成功後，系統獲取當前使用者的「唯一 ID」。

**Step 1. 方案過濾與日期選擇**
1. 系統根據保母在 PRD-001 設定的「預約門禁」規則，動態過濾可見方案：
   - **隱藏制**：若使用者在該方案的「黑名單」中，或不在該方案的「白名單」中，該方案直接不顯示於列表中。
   - **全域封鎖**：若使用者在保母的全域黑名單中，則顯示「保母目前不開放預約」。
2. 飼主選擇「服務方案」，系統自動載入該方案的限制規則 (PRD-003)。
   - **寵物種類驗證**：系統比對該方案支援的「寵物種類」與飼主 Profile (PRD-002) 內現有的毛孩。若飼主沒有對應種類的寵物，系統彈出警告提示：「您沒有適合此方案的寵物，請先至個人檔案新增或選擇其他方案」。
3. 飼主建立「日期群組」選擇服務日期：
   - 支援選擇「單一日期」、「連續區間 (Range)」、或「跳著選多天 (Discrete)」。
   - 系統根據 PRD-003 的「可預訂日期區間」與「生效區間」，自動禁用 (Disable) 不符合方案條件的日期。
4. 針對選定的日期群組，飼主設定「每日服務次數」(例如：一天 2 次)。
5. 支援新增多組「日期群組」，以便在同一張訂單內組合不同的方案或頻率。

**Step 2. 價格試算與提交**
1. 系統執行初步價格計算：
   - 總額 = 各群組 (天數 x 方案單價) 之總和。
2. 飼主確認資料無誤後點擊「送出預約」。
3. 系統執行提交前之最終檢核（最後門禁校驗）。
4. 提交成功後動作：
   - 建立訂單，初始狀態為 `PENDING`。
   - 建立對應的行程項目。
   - 發送「新預約申請」通知給保母。
   - **註：此階段飼主尚未填寫問卷，由保母收單後決定是否指派。**

### 例外處理

| 情境 | 預期行為 |
|------|---------|
| 登入失敗次數過於頻繁 | 系統執行頻率限制 (Rate Limiting)，暫時阻擋驗證請求。 |
| 選擇的日期與方案已額滿 (未來擴充) | 系統提示該日期已額滿，請重新選擇。 |
| 提交訂單時網路中斷 | 前端執行 Retry 機制，並確保後端具備冪等性 (Idempotency) 防止重複建立訂單。 |
| 飼主為黑名單使用者 | 直接封鎖進入預約網頁，不提供後續操作。 |
| 保母尚未完成實名認證 (KYC) | 系統封鎖預約網頁，顯示「此保母尚未完成實名認證，暫不開放預約」。 |
| 所選日期已有其他訂單許候 | 系統不阻擋送出，飼主可正常送單。实際檔期冲突由保母在審核階段 (PRD-006) 決定接受或拒絕。 |

### 資料規則 / 業務規則

- **金額計算與鎖定**：
    - 試算僅供參考，最終報價以保母確認的為準。
    - **單價快照**：一旦訂單確認成交，系統即鎖定當時約定的單價，作為未來任何變更或退款計算的基準。
- **檔期佔用規則**：
    - 檔期不在飼主送單時鎖定，而是在訂單進入 `CONFIRMED` 狀態時才正式佔用。
    - 此設計允許同一時段有多位飼主送單，實際接單與否由保母於審核階段決定。
- **身分識別**：登入憑證應具備時效性。
- **方案卡控**：必須嚴格遵守方案的開放預約與置前天數限制。
- **時區**：預約日期以 `Asia/Taipei` 的「日期 (Date)」為單位。

## 驗收標準（AC）

- [ ] **AC-1 (門禁隱藏)**：若保母針對特定方案設定了白名單，非白名單內的帳號在預約頁面應完全看不到該方案。
- [ ] **AC-2 (全域攔截)**：若帳號在全域黑名單中，進入預約頁面應顯示「不開放預約」而非空白列表。
- [ ] **AC-3 (方案狀態卡控)**：在 Step 1 中，選擇不在方案開放日期內的預約日期時，該方案必須正確顯示為 Disabled。
- [ ] **AC-3 (動態問卷)**：Step 2 呈現的問題內容、類型與必填狀態必須與 PRD-004 的設定完全一致。
- [ ] **AC-4 (金額試算)**：初步試算結果必須與預期邏輯一致。
- [ ] **AC-5 (訂單建立)**：提交後，系統必須正確產生一筆 `PENDING` 訂單，且關聯的「行程」數量與日期完全正確。

## 備註 / 設計限制

- **訪客註冊**：為了降低門檻，訪客在 Step 0 若無帳號，系統應提供快速的 Google OAuth 註冊或免密碼快速註冊流程，以利後續訂單綁定與管理。
- **試算僅供參考**：前端需加註「最終總額由保母評估後確認」之免責聲明。
