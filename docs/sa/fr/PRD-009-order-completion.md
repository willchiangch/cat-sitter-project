## 背景與目的

當一筆訂單內的所有預約行程 (Visits) 皆標記為「已完成」時，該訂單進入結案流程。結案不僅代表服務的正式結束與合約履行完畢，也是保母進行帳務統計的基礎。本模組負責管理訂單的結案邏輯，並提供保母一個直觀的「帳務總覽」列表，以便追蹤經營績效。

## 使用者故事

- 身為 **飼主**，當我確認所有服務已圓滿完成後，我希望能點擊「確認結案」，將訂單歸檔。
- 身為 **系統**，若飼主在所有服務結束後 48 小時內未進行結案或反應問題，我應自動將訂單結案。
- 身為 **保母**，我希望能查閱每月已完成的訂單總額，以便了解我的收入狀況。
- 身為 **保母**，我希望能根據訂單狀態與日期篩選帳務紀錄，方便與匯款資訊對帳。

---

## 功能規格

### 主流程

**A. 飼主確認結案**
1. 訂單內所有行程狀態皆為 `DONE` 時，飼主端詳情頁顯示「確認結案」按鈕。
2. 飼主點擊確認。
3. 系統動作：
   - 訂單狀態更新：`CONFIRMED` -> `COMPLETED`。
   - 記錄日誌 `ORDER_COMPLETED`。
   - 發送結案通知給保母 (含 Email 與站內訊息)。

**B. 系統自動結案 (Auto-Completion)**
1. 系統每日執行排程，檢查符合以下條件的訂單：
   - 狀態為 `CONFIRMED`。
   - 所有行程均已完成 (狀態為 `DONE`)，且最後一個行程完成時間已超過 48 小時。
   - 飼主未於 48 小時內發起任何異議 (非 `DISPUTED` 狀態)。
2. 系統自動執行結案動作，狀態更新為 `COMPLETED`。

**C. 異常行程自動關閉與放行**
1. 若訂單中包含「未執行 (PENDING)」且已超過預計服務日期 72 小時的行程：
   - 系統自動將該行程狀態標記為 `CLOSED_BY_SYSTEM` (系統強制關閉)。
   - 同時視同該行程已處理完畢，允許訂單進入上述自動結案流程。
   - 此動作旨在避免因單一遺漏報告之行程導致保母全額款項無法撥付。

**D. 飼主發起爭議回報 (Report Dispute)**
1. 在所有行程完成後、結案前的 48 小時內，飼主端詳情頁提供「回報問題」按鈕。
2. 飼主點擊後需輸入「問題類別」與「說明內容」。
3. 系統動作：
   - 訂單狀態更新：`CONFIRMED` -> `DISPUTED`。
   - 暫停該訂單的所有自動與手動結案流程。
   - **金流說明**：若為線上支付，系統無法凍結第三方金融機構之自動撥款週期，此狀態僅作為系統內部的帳務標記。
   - 發送緊急通知給保母與系統管理員。

**E. 管理員爭議結案 (Admin Dispute Resolution)**
1. 管理員介入後，不進行對錯裁決，僅作為「共識紀錄者」。
2. 雙方需自行協商結論，並提供證明（如對話截圖、Email、匯款單）給管理員。
3. 管理員於後台：
   - 上傳雙方提供的憑證附件。
   - 根據協商結果調整「最終訂單金額」。
   - 點擊「強制結案」，訂單轉為 `COMPLETED`。

**C. 保母帳務總覽 (Ledger View)**
1. 保母進入「帳務總覽」頁面。
2. 頁面提供「月份篩選」(預設為當月)。
3. 列表呈現：
   - **訂單資訊**：訂單編號、飼主稱呼。
   - **服務期間**：起訖日期。
   - **金額**：最終總金額。
   - **關鍵日期**：付款日 (Payment Date)、結案日 (Completion Date)、預計撥款日 (Payout Date)。
4. 頁面頂端統計該篩選範圍內的「總收入」(依據結案日統計)。

### 例外處理

| 情境 | 預期行為 |
|------|---------|
| 仍有行程尚未完成 (狀態非 DONE) | 系統不顯示「確認結案」按鈕。 |
| 飼主對服務內容有爭議 | (未來擴充) 飼主可點擊「反應問題」，此時系統暫停自動結案邏輯，由管理員或保母介入。 |
| 結案後嘗試修改行程內容 | 系統阻擋。已結案訂單及其關聯行程為「永久唯讀」。 |

### 資料規則 / 業務規則

- **時間軸定義 (Time Dimensions)**：為了統整線上與線下金流的差異，訂單與帳務必須紀錄三個關鍵日期：
  1. **付款日 (Payment Date)**：飼主實際付款的日期 (線下為保母確認入帳日，線上為刷卡成功日)。
  2. **結案日 (Completion Date)**：訂單狀態轉為 `COMPLETED` 的日期，亦即**帳務歸屬日 (Revenue Recognition)**。
  3. **撥款日 (Payout Date)**：保母實際可動用該筆資金的日期。線下轉帳因直接匯入保母帳戶，故 `撥款日 = 付款日`；線上支付則為 `撥款日 = 結案日 + 3 個工作天 (D+3)`。
- **帳務歸屬**：無論付款日或撥款日為何，帳務總覽的「月份營收」皆統一以「結案日」作為歸屬基準。
- **金額精準度**：所有帳務統計必須採 `BigDecimal(18,4)` 處理。
- **爭議處理雙軌制 (Dispute Handling)**：
  - **線上支付 (Online)**：處於 `DISPUTED` 狀態時，系統將**凍結 D+3 撥款倒數**。款項將保留在平台第三方支付專戶中，直到管理員介入標記為「爭議解除」或執行「退款」。
  - **線下支付 (Offline)**：因款項已由保母預收或直接收取，`DISPUTED` 狀態在系統端具有**「申訴存證」**效力。系統會鎖定訂單不可結案，作為保母信用紀錄的扣分依據 (未來擴充)，並引導雙方透過線下方式或管理員調解。
  - **共同限制**：任何 `DISPUTED` 訂單皆不可進入 `COMPLETED` 狀態，且無法在帳務總覽中計入已實現收入。
- **資料保留**：結案後的訂單資料需長期留存供歷史查詢，不隨毛孩刪除而消失。

## 驗收標準（AC）

- [ ] **AC-1 (手動結案)**：飼主點擊確認結案後，訂單狀態正確轉為 `COMPLETED`，且保母端即時更新。
- [ ] **AC-2 (自動結案排程)**：模擬 48 小時後，排程能正確掃描並結案符合條件的訂單。
- [ ] **AC-3 (帳務統計)**：帳務總覽的總收入數字，必須等於該月份所有已結案訂單的最終總金額之和。
- [ ] **AC-4 (唯讀保護)**：處於 `COMPLETED` 狀態的訂單，任何角色皆無法再異動其照片、備註或金額。
- [ ] **AC-5 (篩選邏輯)**：保母切換不同月份時，列表應正確顯示該月份結案的資料。
- [ ] **AC-6 (爭議攔截測試)**：當訂單處於 `DISPUTED` 狀態時，自動結案排程應略過此訂單，且保母端帳務總覽不應計入此筆金額。
- [ ] **AC-7 (撥款凍結測試)**：驗證線上支付訂單在進入爭議狀態後，預計撥款流程應被正確中斷。
