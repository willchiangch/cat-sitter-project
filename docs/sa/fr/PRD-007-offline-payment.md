## 背景與目的

在 MVP 階段，系統優先支援「線下轉帳」模式。保母送出報價後，飼主需將款項匯入保母指定的帳號，並上傳轉帳憑證（如截圖或照片）。經保母人工核對確認入帳後，訂單正式進入「已確認」狀態。此流程能確保在不介接第三方金流的情況下，依然保有完整的交易審核紀錄。

## 使用者故事

- 身為 **飼主**，我希望能看到保母的銀行匯款資訊，以便進行轉帳。
- 身為 **飼主**，我希望能上傳銀行轉帳的截圖，讓保母進行人工核對並啟動預約。
- 身為 **保母**，我希望能檢視飼主上傳的轉帳憑證與後五碼資訊，確認款項入帳後點擊確認以啟動訂單。

---

## 功能規格

### 主流程

**A. 檢視付款資訊 (飼主端)**
1. 飼主進入「待付款訂單」詳情頁。
2. 系統顯示該保母預設的銀行帳號資訊 (帳號、銀行代碼、分行、戶名)。

**B. 上傳憑證 (飼主端)**
1. 飼主完成轉帳後，於頁面點擊「上傳憑證」。
2. 系統強制顯示「線下交易免責聲明」核取方塊：
   - 飼主必須勾選「我瞭解平台不代管此筆線下款項，此筆交易風險由飼主與保母雙方自行承擔」後，方可進行下一步。
3. 填寫資訊：
   - **帳號後五碼**。
   - **上傳圖片**：轉帳成功畫面之截圖或收據照片。
4. 系統動作：
   - 訂單狀態更新：`PENDING_PAYMENT` -> `PAID` (已付清，待核對)。
   - 發送「憑證已上傳」通知給保母 (含 Email 與站內訊息)。

**C. 保母核對憑證**
1. 保母進入訂單詳情，檢視飼主提供的後五碼與憑證圖片。
2. 核對無誤後，保母點擊「確認入帳」。
3. 系統動作：
   - 訂單狀態更新：`PAID` -> `CONFIRMED` (已確認)。
   - 記錄日誌 `PAYMENT_VERIFIED`。
   - 發送「預約已確認」通知給飼主 (含 Email 與站內訊息)。

### 例外處理

| 情境 | 預期行為 |
|------|---------|
| 憑證核對失敗 (金額不符或無效) | 保母點擊「憑證無效」，填寫原因。狀態退回 `PENDING_PAYMENT`，並通知飼主重新處理。 |
| 上傳憑證時檔案過大 | 前端預先壓縮，並提示「憑證圖片需小於 5MB」。 |
| 逾期未付 | (未來擴充) 系統自動將過期未付的訂單轉為 `CANCELLED`。 |

### 資料規則 / 業務規則

- **狀態相依**：只有狀態為 `CONFIRMED` 的訂單，後續的「服務執行面板 (PRD-008)」才會開放。
- **金額儲存與傳輸**：DB 儲存與 API 傳輸一律使用**整數 (INT)**。
- **憑證唯一性**：每一筆訂單在同一時間僅能有一份有效的待核對憑證。
- **免責連結**：飼主完成免責聲明勾選記錄後，將記錄同意時間戳與憑證一併存入訂單，供日後法務作為認知證明。

## 驗收標準（AC）

- [ ] **AC-1 (線下上傳)**：飼主能順利看到匯款資訊，並上傳截圖，送出後狀態正確變更為 `PAID`。
- [ ] **AC-2 (保母確認)**：保母點擊確認入帳後，訂單狀態正確變更為 `CONFIRMED`，且雙方皆收到通知。
- [ ] **AC-3 (憑證無效退回)**：保母退回憑證後，訂單狀態應恢復為 `PENDING_PAYMENT`，且飼主能再次上傳。
- [ ] **AC-4 (編輯保護)**：已確認 (`CONFIRMED`) 的訂單，應禁用「上傳憑證」按鈕。

## 備註 / 設計限制

- **金流安全性**：線下轉帳存在人工誤判風險，系統僅作為資訊對接平台，不承擔實質款項清算責任。
