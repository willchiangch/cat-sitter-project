## 背景與目的

為了建立安全可靠的到府照護平台，確保服務品質與降低潛在風險，所有保母在正式對外接單前，必須經過嚴格的實名認證 (KYC, Know Your Customer)。本模組負責處理保母上傳身分證明文件、管理員審核以及接單資格的連動卡控。

## 使用者故事

- 身為 **保母**，我希望能上傳我的身分證正面與手持身分證自拍照進行實名認證，審核通過後才能正式啟用接單功能。
- 身為 **保母**，我希望能隨時查看我的認證狀態 (審核中/已通過/被退件)，若被退件能得知原因並重新提交。
- 身為 **管理員**，我希望能有一個後台介面審核保母上傳的驗證資料，以把關平台服務品質與安全性。

---

## 功能規格

### 主流程

**A. 保母提交認證資料**
1. 保母初次切換至保母端，或嘗試開啟「開放預約」狀態時，系統攔截並提示「請先完成實名認證」。
2. 進入 KYC 認證頁面，保母需上傳：
   - 身分證正面照片。
   - 手持身分證自拍臉照。
3. 送出後，系統建立一筆 `KYC Record`。
4. 保母檔案的認證狀態轉為 `PENDING_REVIEW` (審核中)。

**B. 管理員後台審核**
1. 管理員登入後台「保母審核清單」。
2. 檢視保母上傳的證件照片與基本資料比對。
3. 執行審核動作：
   - **核准 (Approve)**：該保母的認證狀態轉為 `VERIFIED` (已認證)。
   - **退件 (Reject)**：該保母的認證狀態轉為 `REJECTED` (退件)，管理員需選填退件原因 (如：照片模糊、非本人)。

**C. 審核結果連動**
1. 系統發送審核結果通知 (Email 與站內信) 給保母。
2. 若為核准：保母的預約網頁正式對外生效，可開始接收訂單。
3. 若為退件：保母需重新進入 KYC 頁面重新上傳資料。

### 例外處理

| 情境 | 預期行為 |
|------|---------|
| 圖片檔案過大或格式不符 | 前端限制僅能上傳 JPG/PNG，大小限制 5MB 以下，超過則提示錯誤。 |
| 審核期間嘗試開啟接單狀態 | 系統維持阻擋，顯示「資料審核中，請耐心等候 (約需 1-3 個工作天)」。 |
| 已認證保母被檢舉或違規 | 管理員可於後台強制將狀態改為 `SUSPENDED` (停權)，預約網頁立即失效。 |

### 資料規則 / 業務規則

- **狀態機 (Status Enum)**：
  - `UNVERIFIED` (未認證)：預設狀態。
  - `PENDING_REVIEW` (審核中)：等待管理員處理。
  - `VERIFIED` (已認證)：唯一具備接單資格的狀態。
  - `REJECTED` (退件)：需重新提交。
  - `SUSPENDED` (停權)：強制終止服務資格。
- **接單資格卡控 (KYC Gating)**：在任何狀態下，只要保母的認證狀態不是 `VERIFIED`，其公開的預約網頁應立即失效，飼主點擊時顯示「保母尚未完成實名認證」。
- **個資保護**：身分證照片上傳至 GCP Bucket 時必須設定嚴格的存取權限 (Private ACL)，僅限系統後台與特定管理員可讀取，嚴禁對外公開。

## 驗收標準（AC）

- [ ] **AC-1 (資料提交)**：保母能順利上傳符合格式的照片並送出，送出後狀態必須即時更新為 `PENDING_REVIEW`。
- [ ] **AC-2 (後台審核)**：管理員能在後台執行核准或退件操作，且保母端會收到對應的通知。
- [ ] **AC-3 (退件重提)**：被退件的保母能看到退件原因，並允許重新上傳資料發起新的 KYC 請求。
- [ ] **AC-4 (資格卡控)**：模擬飼主進入一個非 `VERIFIED` 狀態保母的預約網頁，必須被系統完全阻擋並顯示適當訊息。
- [ ] **AC-5 (安全防護)**：嘗試直接透過 URL 存取其他保母上傳的身分證圖片網址，必須回傳 `403 Forbidden`。

|------|------|
| 資料模型變更 | 是。需獨立建立 `KYC Record` 表，並擴充 Profile 的 `approval_status`。 |
| API 新增/修改 | 是。新增 KYC 上傳 API、管理員審核 API。 |
| 前端頁面 | 是。新增保母端的認證上傳頁面、管理員後台的審核列表與詳情頁。 |
| 通知/郵件 | 是。審核通過與退件皆需發送通知。 |
