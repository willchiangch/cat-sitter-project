## 背景與目的

為了維持「貓咪保母 PWA」的 MVP 開發極簡性，我們在架構上刻意排除了多項複雜的自動化系統（例如：自動化證件辨識、自動化折扣碼引擎、全自動退款流程）。取而代之，本模組負責定義一個基礎的「系統管理後台」，提供管理員（營運人員）一個集中的介面，以人工介入的方式處理系統的例外邊界與營運核心工作。

## 使用者故事

- 身為 **系統管理員**，我希望能查閱保母上傳的 KYC 證件，並手動核准其接單資格。
- 身為 **系統管理員**，當營運需要給予特定保母優惠（如早鳥免月費）時，我能直接手動調整他的方案等級與到期日，不需透過工程師改資料庫。
- 身為 **系統管理員**，當收到飼主的訂單爭議投訴時，我能查閱該筆訂單的歷史紀錄，並手動執行退款或解鎖款項。
- 身為 **系統管理員**，我希望能動態維護「敏感詞庫」，讓系統自動攔截違規的保母自介內容。

## 功能規格

### 主流程

**B. 保母與訂閱方案管理 (對應 PRD-012, PRD-017)**
1. **KYC 審核**：核准或退件保母的接單資格。
2. **營運配發/人工展期**：管理員可手動覆寫帳號的方案等級與到期日。
3. **帳號狀態控管與暫停條款**：
   - 可執行「警告、停權、註銷」。
   - **停權處理**：當保母被停權時，系統自動「凍結」其訂閱方案倒數。
   - **復權處理**：復權後系統自動依據停權天數「延長」方案有效期，不損害保母已支付權益。
   - **註銷處理**：若執行帳號註銷，系統自動計算剩餘天數之應退款金額，產生退款工單。

**C. 爭議訂單處理 (對應 PRD-009)**
1. 列表呈現所有 `DISPUTED` 狀態訂單。
2. **管理員定位**：不進行對錯裁決，僅作為「雙方達成共識後」的執行者。
3. **操作介面**：
   - 憑證存檔區：上傳雙方協商後的結論憑證（如對話截圖、Email、匯款單）。
   - 金額調整區：根據協商結果手動調整最終應付金額。
   - 強制結案區：執行後系統轉為 `COMPLETED`。
4. **金流說明**：線上支付由金融機構自動撥款，系統無法中途凍結。若判定需退款，管理員依據判定執行「API 退刷」或「產出退款提醒」。

**E. 內部信用指標管理 (Internal Trust Score)**
1. **基準定義**：每位保母預設指標為 100 點。
2. **手動獎懲**：管理員可針對保母行為進行點數增減：
   - **獎勵**：如協助處理緊急轉介、獲得多筆優質評價（手動鼓勵）。
   - **懲罰**：如訂單爭議判賠、回覆率極慢、線下交易違規等。
3. **隱私規則**：此資訊為「後台內部指標」，**飼主與保母在前台皆無法查看**。
4. **連動預警**：若點數低於設定門檻（例如 60 點），管理後台列表應標註為「高風險」，提醒管理員評估是否執行強制隱藏 (PRD-018) 或撤銷認證 (PRD-017)。

**D. 敏感詞庫維護 (對應 PRD-018)**
1. 提供簡易的 CRUD 介面，讓管理員新增、編輯、刪除敏感/違規關鍵字 (例如：皮下、注射、醫療)。
2. 該詞庫即時生效於保母公開檔案的攔截器中。

### 例外處理

| 情境 | 預期行為 |
|------|---------|
| 管理員操作敏感金流或權限 | 系統強制所有異動操作（如：刷退、變更方案、核准 KYC）都必須留下 Audit Log（誰、什麼時間、改了什麼、原因為何）。 |
| 嘗試退款金額大於訂單總額 | 系統阻擋並提示「退款金額不得大於原訂單總金額」。 |

### 資料規則 / 業務規則

- **存取限制**：管理後台必須實作獨立的登入入口（如 `/admin`），且僅允許被授權的特定內部帳號存取，嚴禁開放外部註冊。
- **免除折扣碼系統**：本系統**不實作**任何折扣碼或序號兌換邏輯，所有推廣與補償皆透過「B. 訂閱方案與效期管理」人工處理。

## 驗收標準（AC）

- [ ] **AC-1 (KYC 審核)**：管理員能在後台成功將待審核的保母轉為 `VERIFIED`，且保母端即時生效。
- [ ] **AC-2 (人工展期)**：管理員能無縫將某位 Free 版保母手動升級為 Pro 版並設定 30 天效期，且系統日誌確實記錄異動原因。
- [ ] **AC-3 (退款執行)**：針對線上支付的爭議訂單，管理員能執行部分或全額刷退，且訂單狀態能正確結案。
- [ ] **AC-4 (敏感詞生效)**：管理員在後台新增的敏感詞，能立即阻擋保母在前台的違規自我介紹儲存。
- [ ] **AC-5 (內部信用指標異動)**：管理員執行指標點數增減後，保母資料中的指標應正確更新，且系統日誌記錄該筆異動與原因。

## 影響範圍

| 面向 | 說明 |
|------|------|
| 資料模型變更 | 是。需新增 `Admin` 實體與 `System Audit Log` 實體。 |
| API 新增/修改 | 是。需建立專屬於管理員操作範圍的 `/api/admin/*` 端點。 |
| 前端頁面 | 是。需開發獨立的管理後台 SPA 或使用現成低代碼框架 (如 Retool)。 |

## 備註 / 設計限制

- **金流手續費揭露**：保母在開啟「線上支付」功能時，系統必須明確告知金融機構收取的交易手續費率。平台方對服務收入維持「零抽成」。
- **管理員權限與審計**：所有異動操作必須記錄於 `AUDIT_LOG`。
