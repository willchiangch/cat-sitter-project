## 背景與目的

為了提升交易效率與保障雙方權益，本平台提供線上支付功能。除了用於保母購買 SaaS 訂閱方案外，也提供飼主支付照護費用的管道。針對照護費用，平台採取代收代付模式，確保保母在完成服務後能穩定獲得報酬，並透過明確的撥款週期規範，統一帳務處理流程。

## 使用者故事

- 身為 **保母**，我希望能透過平台收取訂單費用，減少線下催款或對帳的麻煩。
- 身為 **保母**，我希望能清楚知道每一筆服務報酬何時會撥入我的銀行帳戶。
- 身為 **保母**，當我升級方案時，我希望能選擇信用卡或虛擬帳號支付，並在付費後立即生效。
- 身為 **飼主**，我希望能使用信用卡線上支付預約費用，提升下單效率。

## 功能規格

### 1. 支付場景與模式

系統區分兩大類支付行為，其帳務處理邏輯不同：

#### A. 平台 SaaS 訂閱支付
- **適用範圍**：保母購買 Basic / Pro / Premium 方案。
- **支付方式**：支援信用卡一次性支付、ATM 虛擬帳號轉帳。
- **生效邏輯**：付費成功後，系統應立即更新保母之權限等級，並從當日起算新的合約週期。

#### B. 照護服務費支付 (代收代付)
- **適用範圍**：飼主支付預約單費用。
- **支付方式**：信用卡。
- **流程規則**：
  1. **預付**：飼主在保母確認報價後完成支付，款項暫由平台代管。
  2. **完工**：服務執行結束且完成「結案」動作 (PRD-009)。
  3. **結算與撥款**：訂單結案後，系統進入撥款作業流程。

#### C. 保母銀行帳戶綁定 (Payout Account)
1. 保母進入「收款設定」。
2. 系統顯示「線上支付」開關，並於旁邊明確標註：**「開啟此功能後，每筆交易將由金融機構扣除約 X.X% 之手續費，系統本身不收費。」**
3. 保母填寫銀行收款帳戶並確認開啟。
   - 銀行代碼 (下拉選單)。
   - 分行代碼 / 名稱。
   - 帳號。
   - 戶名 (需與 KYC 證件名稱一致)。
3. 系統執行基礎格式驗證 (不執行即時入戶驗證)。
4. 綁定成功後，該帳戶將作為 `PRD-015` 撥款流程的目標帳號。

### 2. 撥款時程規範 (Payout Cycle)

針對「照護服務費」，平台採用的撥款時程定義如下：

- **撥款基準日**：訂單狀態轉為「已結案 (DONE)」的日期為 T 日。
- **入帳日期**：T + 3 個工作日 (暫定)。
- **撥款金額**：訂單原始總額扣除平台服務費 (若有) 與轉帳手續費後之淨額。
- **查詢機制**：保母可在帳務面板查看「預計撥款日」與「撥款狀態 (待撥款/已撥款)」。

### 3. 退款規則

#### A. 照護服務費退款 (相依 PRD-016)
- **主動取消**：若訂單在服務開始前被取消且符合退款條件，系統應將款項退還至原信用卡。
- **部分退款**：若因服務天數縮減 (PRD-016) 產生價差，系統應依據重算後的金額執行退款。
- **手續費吸收歸屬**：因變更或取消產生的「金流代收手續費損失」，由**平台全額吸收**。飼主應獲得差額之全額退款，不需負擔退款手續費。

#### B. SaaS 訂閱退款 (人工處理流程)
- **申請機制**：保母於訂閱效期內欲終止服務並退款時，需透過客服信箱或系統回報管道提出「人工退款申請」。
- **管理員操作**：
  1. 管理員於後台檢核保母的使用紀錄與剩餘效期。
  2. **金額折算**：管理員手動輸入應退金額 (依據使用天數或特定折價規則)。
  3. **執行退款**：
     - 若原支付方式為信用卡：管理員透過金流後台執行「信用卡刷退」(全額或部分)。
     - 若原支付方式為虛擬帳號/轉帳：管理員執行線下匯款，並於系統標註「已手動退款」。
  4. **權限回收**：退款完成後，系統應手動或自動將保母之訂閱等級調降回「體驗版 (Free)」。

## 業務規則與限制

- **金額計算與儲存**：
  - **資料儲存與傳輸**：DB 儲存與 API 傳輸一律採用 **整數 (INT)**，以符合台灣市場 (TWD) 無小數之慣例。
  - **高精度運算**：後端在處理手續費分帳、折扣計算時，應轉為 `BigDecimal` 進行，最終入庫或呈現前採 `RoundingMode.HALF_UP` 轉回整數。
- **唯一性**：每一筆訂單應有唯一的交易流水號，避免重複支付。
- **支付路徑與狀態機差異**：
  - **線上支付路徑**：`PENDING_PAYMENT` -> (刷卡成功回傳) -> `CONFIRMED`。系統自動跳過 `PAID` (待核帳) 狀態，直接確認訂單。
  - **線下支付路徑 (PRD-007)**：`PENDING_PAYMENT` -> (上傳憑證) -> `PAID` -> (保母手動確認入帳) -> `CONFIRMED`。
- **手動續約原則**：(PRD-012) 本平台不提供信用卡自動扣款續約功能。所有付費行為皆由使用者主動發起支付請求。
- **撥款資料**：保母必須完成銀行帳戶資訊的綁定，否則系統將暫緩該保母的所有撥款作業，直到資料補齊。
- **退款爭議**：任何人工退款均需附註「退款原因」，以利後續對帳與稽核。

## 驗收標準（AC）

- [ ] **AC-1 (付費生效)**：保母完成 SaaS 方案支付後，其功能權限必須在 3 分鐘內自動解鎖，不需人工審核。
- [ ] **AC-2 (狀態連動)**：飼主完成信用卡支付後，訂單狀態必須自動從「待付款」轉變為「已付款」。
- [ ] **AC-3 (撥款紀錄生成)**：當訂單點擊「結案」時，系統必須自動產生一筆對應的「撥款紀錄」，且預計撥款日計算正確 (T+3)。
- [ ] **AC-4 (手續費試算)**：確認撥款淨額是否已正確扣除預設的平台手續費。
- [ ] **AC-5 (虛擬帳號有效期)**：若選擇虛擬帳號支付方案，逾期未轉帳則該筆交易自動失效，權限不予開通。
- [ ] **AC-6 (管理員人工退款)**：驗證管理員能在後台輸入特定退款金額，執行刷退動作後，保母的訂閱狀態必須正確降級回體驗版。
- [ ] **AC-7 (銀行帳戶綁定驗證)**：保母在未填寫銀行帳戶前，系統應在帳務總覽頁面顯示「缺少撥款資訊」警告，且該狀態下的訂單無法進入撥款排程。
