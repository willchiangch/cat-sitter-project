## 背景與目的

專業保母通常需要同時管理多個案子與個人私人行程。為了避免漏接訂單或產生時段衝突，系統需提供自動同步機制，將系統內「已確認」的訂單行程自動推送到保母常用的個人行事曆 (如 Google Calendar) 中。這不僅提升了作業效率，也能確保保母隨時隨地掌握最新的工作行程。

## 使用者故事

- 身為 **保母**，我希望能將系統內的服務行程同步到我的 Google 行事曆，以便我能一併檢視工作與私人時間。
- 身為 **保母**，當訂單發生日期異動 (PRD-016) 或取消時，我希望系統能自動更新我的 Google 行事曆，不需要手動調整。
- 身為 **保母**，當我在 Google 行事曆上把全天事件拖曳到特定時段 (如 14:00-15:00) 時，我希望系統的「今日行程」也能自動根據這個時間排序。
- 身為 **保母**，當同步授權失效時，我希望能收到提醒以便重新綁定。

## 功能規格

### 主流程

**A. 第三方行事曆綁定 (OAuth)**
1. 保母進入「設定 > 行事曆同步」頁面。
2. 點擊「連結 Google 行事曆」。
3. 系統導向 Google OAuth 2.0 授權頁面，請求 `Calendar.Events` 讀寫權限。
4. 授權成功後，系統記錄 Access Token 與 Refresh Token。

**B. 初次同步與建立專屬日曆**
1. 系統自動在保母的 Google Calendar 中建立一個名為「貓咪保母 - [系統名]」的專屬日曆。
2. 系統掃描該保母名下所有狀態為 `CONFIRMED` 或 `IN_PROGRESS` 的訂單。
3. **事件映射原則 (1 Visit = 1 Event)**：
   - 系統並非以「訂單 (Order)」為單位同步，而是以「單次行程 (Visit)」為單位進行同步。
   - **多日/跳日預約**：若飼主預約 1 號與 5 號，系統會建立兩個獨立的單日事件，不會產生橫跨 1~5 號的長區間事件，避免佔用保母版面。
   - **單日多次預約**：若飼主預約單日 2 次，系統會在同一天建立「兩個獨立的」全天事件。
4. 將這些行程 (Visit) 推送到專屬日曆：
   - **時間設定**：預設為**全天事件 (All-Day Event)**。
   - **標題**：`[貓保母] 飼主稱呼 - 毛孩名 (第N次)`
   - **內容**：訂單連結、照護注意事項摘要。

**C. 自動事件驅動同步 (Real-time Sync)**
1. 當訂單發生以下狀態變動時，系統自動觸發同步：
   - **訂單確認 (CONFIRMED)**：建立新事件。
   - **日期變更 (MODIFIED)**：更新既有事件的起訖日期。
   - **訂單取消 (CANCELLED)**：刪除對應的行事曆事件。
   - **服務完成 (DONE/COMPLETED)**：透過局部更新 (PATCH) 將標題加上 [已完成]。

**D. 雙向時間同步 (Two-way Time Sync)**
1. 系統需訂閱該專屬日曆的 Webhook 推播 (Google Calendar Push Notifications)。
2. 當保母在外部行事曆中將「全天事件」改為「特定時段」時：
   - 系統接收到通知後，自動更新該次行程的「預計執行時間」。
   - **通知觸發**：系統同步更新時間後，必須自動發送一則通知給飼主，告知「保母已更新預計抵達時間」，確保雙方資訊對稱。
   - 例如：保母將「單日 2 次」的兩個全天事件，分別拖曳到 08:00 與 18:00，系統內的兩個行程就會分別更新時間並通知飼主。
   - `PRD-008` 的「今日行程」列表將自動依據此時間序進行排序。
3. **單向狀態防護**：系統僅接收「時間」的異動，若保母在 Google Calendar 中「刪除」該事件，系統不作狀態回滾，仍以系統內的合約狀態為準。

**E. 手動強制同步與修復 (Manual Sync & Recovery)**
1. 保母可在「行事曆同步」頁面點擊「立即同步 / 檢查遺漏」。
2. 系統啟動比對機制：掃描所有 `CONFIRMED` 與 `IN_PROGRESS` 的訂單。
3. 若發現系統內的訂單在 Google Calendar 中對應的 Event ID 已失效或被保母「誤刪」，系統將自動**重新建立該事件**，並更新系統內的 Event ID 映射紀錄。
4. 若資料有落差，一律以系統當前狀態強制覆蓋 (PATCH) 至 Google Calendar。

### 例外處理

| 情境 | 預期行為 |
|------|---------|
| 授權逾期 (Token Expired) | 系統嘗試使用 Refresh Token 自動刷新；若失敗則發送通知請保母重新登入授權。 |
| Google API 配額限制或服務中斷 | 系統將同步任務放入「失敗重試佇列」，待恢復後自動補齊。 |
| 保母手動在 Google Calendar 誤刪事件 | 系統狀態不受影響。保母可透過上述「手動強制同步 (E)」機制重新建立該事件。 |

### 資料規則 / 業務規則

- **時間雙向 / 狀態單向**：這是一個混合型同步機制。「時間」欄位允許雙向同步 (保母可透過 Google Calendar 安排時段並同步回系統)；但「訂單狀態」為單向同步 (系統 -> 行事曆)。
- **資料範圍**：僅同步狀態為 `CONFIRMED` 或 `IN_PROGRESS` 的正式行程。
- **隱私安全性**：同步至第三方行事曆的資訊應過濾敏感個資 (如：僅顯示飼主稱呼，不顯示完整住址或電話，建議放訂單連結)。

## 驗收標準（AC）

- [ ] **AC-1 (授權綁定)**：保母能順利完成 Google OAuth 授權流程，且系統能正確取得 Token。
- [ ] **AC-2 (事件建立)**：當一筆訂單轉為 `CONFIRMED` 時，保母的 Google Calendar 應在 5 分鐘內出現對應事件。
- [ ] **AC-3 (異動連動)**：根據 PRD-016 修改訂單日期後，行事曆上的事件時間應同步更新。
- [ ] **AC-4 (取消刪除)**：訂單取消後，對應的行事曆事件應被正確移除。
- [ ] **AC-5 (錯誤恢復)**：當網路暫時中斷導致同步失敗時，恢復後系統應能自動補傳。
- [ ] **AC-6 (實體映射驗證)**：驗證當飼主送出「跳日預約」或「單日兩次」的訂單並確認後，行事曆必須精準產生對應數量之「獨立」全天事件，嚴禁合併為橫跨多日的單一事件。
