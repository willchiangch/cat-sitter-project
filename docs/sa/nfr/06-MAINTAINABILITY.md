## 1. 程式碼品質與測試規範

**思維：高品質的程式碼是系統穩定與快速迭代的基石。**

*   **程式碼開發規範**：前後端開發必須遵循團隊統一的風格規範，並透過自動化工具（Lint / Formatter）進行強制校對，確保不同開發者的產出具備一致的可讀性。
*   **核心邏輯自動化測試**：
    *   **單元測試 (Unit Test)**：針對核心業務邏輯（如：價格計算、狀態轉換）之單元測試覆蓋率應達 **80%**。
    *   **關鍵元件測試 (Component Test)**：針對具備複雜互動邏輯的前端元件（如：日曆選擇器、任務勾選面板、多媒體上傳組件）應編寫元件測試，確保 UI 狀態切換之正確性。
    *   **API 整合測試**：關鍵 API 需具備自動化整合測試，確保資料庫互動正確。
    *   **UI E2E 測試**：MVP 階段暫緩強制自動化 E2E，待介面穩定後再導入。元件測試將作為此階段的主要 UI 驗證手段。

## 2. 營運與業務監測 (Observability)

**思維：系統必須具備透明度，發生問題時能快速追蹤業務現場。**

*   **服務存取紀錄**：所有對外服務請求均需自動紀錄處理時間、執行狀態與耗時，以便進行效能瓶頸分析。
*   **業務稽核日誌 (Business Logs)**：
    *   針對關鍵操作（如：訂單成立、結案、付款成功、退款執行），系統必須產生結構化的稽核日誌。
    *   **日誌內容**：必須包含操作者身分、時間戳記、變更前後的關鍵值對比，確保具備追溯歷史的能力。
*   **異常即時通知**：當生產環境偵測到嚴重的系統錯誤（5xx 錯誤）或核心流程中斷時，系統應具備自動化通知機制（如：發送至管理員 Email 或通訊軟體頻道）。

---

## 3. 自動化管理流程

**思維：減少人工介入，降低手動部署造成的錯誤風險。**

*   **自動化發佈流程 (CI/CD)**：系統需建立自動化構建與發佈流程。程式碼經由測試驗證通過後，應能一鍵部署至測試或生產環境，確保發佈內容與測試內容一致。
*   **資料結構版本管理 (DB Migration)**：所有的資料結構異動必須透過版本化的管理工具執行，嚴禁直接手動操作生產環境資料結構，確保資料庫異動可回溯、可追蹤。

---

## 4. 技術債約束

*   **套件版本管理**：嚴禁使用已停止維護 (Deprecated) 的第三方函式庫。
*   **架構文件同步**：任何對核心邏輯的修改，均需同步更新對應的 SA/SD 文件，確保文件與程式碼一致。
